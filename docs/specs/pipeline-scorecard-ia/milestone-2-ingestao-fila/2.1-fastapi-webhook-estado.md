## Contexto
O pipeline é orientado a eventos. Precisamos expor um ponto de entrada HTTP público (webhook) para receber notificações de gravações prontas. Esse webhook irá iniciar de forma assíncrona a esteira de processamento da gravação, mantendo o controle de estado e permitindo que sistemas terceiros acompanhem o andamento.

## Objetivo
Criar uma aplicação FastAPI contendo a rota do webhook de entrada, validação do payload, persistência inicial do estado no PostgreSQL e enfileiramento do job inicial no Redis Queue (RQ).

## Especificação Técnica
- **FastAPI Setup:**
  - Configurar a estrutura básica da aplicação FastAPI.
  - Implementar middleware de tratamento global de exceções.
- **Endpoint de Webhook (`POST /webhooks/recording`):**
  - Receber payload JSON contendo:
    - `recording_url`: String (caminho local absoluto ou URL do arquivo de áudio gerado pelo gerador sintético).
    - `job_id`: String (identificador correspondente aos arquivos de vaga/checklist do Milestone 1).
  - Fluxo do endpoint:
    1. Validar a existência do arquivo/recurso de áudio.
    2. Persistir no PostgreSQL um novo registro da tabela `Interview` com status `recebida`.
    3. Despachar um job para o Redis Queue (RQ) com o `interview_id`.
    4. Retornar status HTTP 202 (Accepted) com o `interview_id` e o `status` atual.
- **Endpoint de Consulta (`GET /interviews/{interview_id}`):**
  - Retornar o estado atualizado do registro correspondente no PostgreSQL, incluindo timestamps, transcrição (se houver), e scorecard (se houver).

## Critérios de Aceite
- [ ] O servidor FastAPI inicia corretamente na porta padrão (ex: `8000`).
- [ ] Enviar uma requisição POST válida para `/webhooks/recording` cria a linha no Postgres e retorna HTTP 202.
- [ ] O enfileiramento no Redis é acionado (pode ser verificado simulando a conexão com Redis).
- [ ] Enviar payloads inválidos retorna HTTP 400 ou 422 sem quebrar o servidor.

### Testes Automatizados
- [ ] Criar testes utilizando `fastapi.testclient.TestClient` para o webhook e rota GET.
- [ ] Usar mocks para simular a fila Redis (`mock` ou biblioteca de mock de RQ) para verificar que o job de processamento foi despachado com os parâmetros corretos.
