# Card 2.2 — Worker RQ & Resiliência de Estado

## Contexto
O processamento de áudios longos e chamadas de LLM é demorado. Precisamos de um processador em segundo plano baseado em fila (Redis + RQ). A resiliência é um ponto central: se o servidor ou o container do worker cair no meio do fluxo, ao ser reiniciado o pipeline deve continuar de onde parou (utilizando o PostgreSQL como fonte da verdade dos dados salvos) sem a necessidade de reprocessar tarefas computacionalmente custosas, como a transcrição.

## Objetivo
Implementar o loop do Worker RQ, a função orquestradora da esteira de processamento e a lógica de recuperação de falhas (resiliência de estado).

## Especificação Técnica
- **Orquestração da Esteira (`process_interview`):**
  - Implementar a função principal executada pelo worker: `process_interview(interview_id: UUID)`.
  - O fluxo lógico deve verificar a coluna `status` da entrevista no PostgreSQL para decidir o ponto de partida:
    1. Se `status == "recebida"`: atualizar status para `transcrevendo`, rodar transcrição e salvar o texto em `transcription_raw`.
    2. Se `status == "transcrevendo"` (mas já possuir dados de transcrição salvos por execução anterior incompleta) ou se `status == "diarizada"` pendente de merge: pular transcrição física e passar para a etapa de diarização. Atualizar para `diarizada` e salvar o JSON resultante em `diarization_raw`.
    3. Se `status == "diarizada"` (com dados de transcrição e diarização preenchidos): atualizar status para `pontuando`, rodar a engine de scoring e salvar no campo `scorecard`.
    4. Se `status == "pontuando"` (e o scorecard foi gerado com sucesso): atualizar status para `aguardando_aprovacao` e acionar a notificação de aprovação.
- **Estrutura do Worker:**
  - Criar um script de entrada (`entrypoint.sh` ou similar) para iniciar o worker do RQ conectado ao Redis configurado.
  - Tratar exceções gerais: se o processamento lançar erro, salvar o stack trace em um campo de logs ou meta da entrevista e manter o último estado persistido íntegro para tentativas futuras.

## Critérios de Aceite
- [ ] O comando `rq worker` conecta ao Redis e aguarda jobs sem erros.
- [ ] O fluxo executa sequencialmente as etapas e atualiza a coluna `status` no PostgreSQL.
- [ ] Se o processo for interrompido manualmente após a escrita de `transcription_raw` e reiniciado, o pipeline retoma diretamente a partir da etapa seguinte (diarização ou scoring) sem invocar o motor de transcrição de áudio novamente.
